Architecture Overview

Detailed information of our technical contributions in Escape from Uni.

Debugging Tools

What: It displays green hitbox lines, grid lines, a small box at the player’s feet, an interaction circle and a center X so collision and alignment areas are visible on screen.

Why: It helps us debug faster and fix issues quickly by showing green hitbox lines, grid lines, a small box at the player’s feet, an interaction circle and a center X.

Evidence: View PDF

Rendering System

What: The system renders objects in the correct order based on layer and Y-position, with a camera that smoothly follows the player.

Why: This ensures correct occlusion and a consistent top-down view.

Editor

What: EntityMaker creates game objects and attaches components such as sprite, physics, animation, and interactions.

Why: It reduces boilerplate and speeds up level creation.

Testability

What: Systems such as RenderOrderComparator, WorldCameraSystem, EntityMaker, and MazeGame are independent.

Why: This allows isolated testing and easier maintenance.

Modularity (Extreme Decoupling)

What: Rendering, camera, physics, and entity creation are separated into independent systems.

Why: This keeps the architecture flexible and maintainable.

Evidence: View PDF

Interaction System

What: Detects interactions and sends messages to objects.

Why: Enables reusable, data-driven interactions.

Assets

What: Assets are managed centrally via AssetLoader, AssetId, and AssetPaths.

Why: Simplifies access and reduces path errors.

Evidence: View PDF

Fixed time-step engine

What: Uses a fixed 1/60s update step.

Why: Keeps physics and logic consistent across machines.
